# Educational Review: Phase 1 - Backend Statistics Calculation & APIs

## Введение
Первая фаза реализации трека «Leaderboards & Stats» была посвящена созданию мощного аналитического движка на бэкенде. Мы прошли путь от определения структур данных до реализации высокопроизводительных запросов к базе данных.

## Ключевые архитектурные концепции

### 1. Проектирование DTO и использование Java Records
Мы использовали **Java Records** для всех объектов передачи данных (DTO).
- **Почему:** Records неизменяемы по умолчанию, имеют встроенные методы `equals`, `hashCode` и `toString`, что идеально подходит для данных статистики.
- **Урок:** Использование `@Builder` вместе с Records (через Lombok) позволяет сохранять чистоту кода при создании сложных вложенных структур, таких как `PlayerStatsResponse`.

### 2. Производительность: Java vs SQL
Одной из главных дилемм фазы был выбор места агрегации данных.
- **Первоначальный подход:** Расчет статистики в памяти Java (метод `calculateStatsFromMatches`).
- **Финальное решение:** Использование **Native SQL Projections**.
- **Почему:** База данных (PostgreSQL/H2) оптимизирована для группировки и подсчета тысяч строк. Передача только итоговых цифр в приложение экономит пропускную способность сети и Heap-память.
- **Техника:** Мы применили **CTE (Common Table Expressions)** с `WITH` для структурирования сложных запросов и `UNION ALL` для объединения данных об игроках из разных колонок (нападающий/защитник).

### 3. Принцип Tell, Don't Ask (Проси, а не спрашивай)
В ходе ревью мы перенесли логику определения ролей и побед из сервиса (`StatisticsService`) внутрь сущности `Match`.
- **До:** Сервис спрашивал ID игроков у матча и сам решал, кто выиграл.
- **После:** Сервис вызывает `match.isWinner(player)`.
- **Результат:** Инкапсуляция бизнес-правил внутри доменных объектов делает код более читаемым и защищенным от изменений правил игры.

## Практические приемы и "грабли"

### Совместимость H2 и PostgreSQL
Мы столкнулись с проблемой проекции UUID в строковый формат в Native SQL при использовании H2.
- **Решение:** Использование `CAST(u.id AS VARCHAR)` в SQL и `UUID.fromString()` в Java.
- **Урок:** Нативные запросы всегда требуют проверки на всех поддерживаемых типах БД.

### Важность Fail-Fast
Добавление `Objects.requireNonNull()` в сервисные методы на этапе ревью позволило гарантировать, что ошибки будут обнаружены в момент вызова, а не глубоко внутри сложной логики запроса.

Бэкенд статистики теперь готов к масштабированию. Мы заложили фундамент, который выдержит тысячи матчей и обеспечит мгновенный доступ к рейтингам. В следующей фазе мы перейдем к визуализации этих данных на фронтенде.
