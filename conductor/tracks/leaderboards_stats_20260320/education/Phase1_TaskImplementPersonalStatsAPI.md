# Educational Log: Phase 1 - Backend Statistics Calculation & APIs
## Task: Implement Personal Statistics API

### Введение
Для предоставления игрокам глубокой аналитики нам необходимо реализовать API личной статистики. Это включает в себя не только общие показатели (победы/поражения), но и разбивку по ролям (нападающий/защитник), а также Head-to-Head (H2H) статистику — результаты игр против каждого уникального оппонента.

### Резюме шага: Реализация и оптимизация (Green Phase)
Мы успешно реализовали эндпоинты для персональной и H2H статистики. В ходе работы мы применили несколько продвинутых архитектурных приемов.

#### Ключевые решения:
1.  **Агрегация на стороне БД (Projections)**: Изначально персональная статистика считалась в памяти Java. В ходе код-ревью мы выявили риск масштабируемости и переписали логику на Native SQL запросы с использованием интерфейсных проекций (`PlayerStatsProjection`). Это позволяет базе данных возвращать уже готовые итоговые цифры, экономя память и время.
2.  **Tell, Don't Ask**: Мы перенесли логику проверки ролей (`isAttacker`, `isDefender`, `isWinner`) непосредственно в сущность `Match`. Сервис больше не "спрашивает" у матча ID игроков, чтобы принять решение, а "приказывает" матчу проверить участие конкретного пользователя.
3.  **H2H через CTE**: Для Head-to-Head статистики мы использовали Common Table Expressions (CTE), чтобы сначала собрать всех возможных оппонентов игрока из разных команд, а затем агрегировать их в финальный результат.

#### Педагогическая заметка:
Обратите внимание на `PlayerStatsProjection`. Использование интерфейсов для маппинга Native SQL — это очень мощный и чистый инструмент в Spring Data JPA. Нам не нужно создавать классы-обертки вручную, Spring сам создаст прокси-объект на основе имен геттеров.

### Итоги задачи:
Персональная аналитика готова. Теперь игроки могут видеть свои сильные стороны в разных ролях и историю противостояний с конкретными соперниками.
