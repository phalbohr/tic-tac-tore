# Educational Log: Phase 1 - Backend Statistics Calculation & APIs
## Task: Implement Global Leaderboard API

### Введение
На этом этапе мы реализуем API для получения глобальных рейтингов (лидербордов). Это требует эффективной агрегации данных по всем игрокам, поддержки пагинации и фильтрации.

### Резюме шага: Проектирование API (Red Phase)
Перед реализацией контроллера и сервиса мы создали интеграционный тест `StatisticsIntegrationTest`. Это позволило нам определить контракт API и убедиться, что он возвращает ожидаемые данные.

### Резюме шага: Реализация фильтрации (Green Phase)
Мы расширили API лидерборда, добавив поддержку фильтров по ролям (Attacker/Defender), временным окнам (неделя, месяц, год) и минимальному порогу матчей.

#### Ключевые решения:
1.  **Динамический SQL через параметры**: Мы использовали именованные параметры в Native SQL (`:type`, `:since`, `:minMatches`). Это позволяет базе данных повторно использовать план выполнения запроса, повышая безопасность и производительность.
2.  **Обработка ролей в UNION**: Чтобы фильтровать по ролям (нападающий/защитник), мы добавили литеральную колонку `role` в каждый блок `UNION ALL`. Это позволило во внешнем запросе использовать простое условие `WHERE role = :type`.
3.  **Логика временных окон**: Расчет даты начала периода вынесен в сервис. Использование `switch` (Java 17+) делает код чистым и легко расширяемым. Мы используем `minusWeeks(1)`, `minusMonths(1)` и т.д. для динамического расчета.
4.  **Фильтрация через `HAVING`**: Для исключения игроков с малым количеством матчей мы применили `HAVING COUNT(*) >= :minMatches`. Это стандартный способ фильтрации агрегированных данных.

#### Педагогическая заметка:
Обратите внимание на `countQuery` в аннотации `@Query`. Для Native SQL с пагинацией Spring Data JPA не может автоматически сгенерировать запрос для подсчета общего количества элементов. Мы должны написать его вручную, обеспечив полную идентичность условий фильтрации с основным запросом.

### Итоги задачи:
API лидерборда полностью соответствует спецификации. Мы обеспечили высокую производительность за счет агрегации на стороне БД и гибкость за счет системы фильтров.
