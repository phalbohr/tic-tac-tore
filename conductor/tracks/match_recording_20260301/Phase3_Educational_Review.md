# Образовательная глава: Разработка Match Recording UI (Фаза 3)

## Введение
В этой главе мы рассмотрели процесс создания сложного пользовательского интерфейса для записи результатов матчей 2v2 (Doubles). Основное внимание было уделено TDD, управлению состоянием и реализации специфических бизнес-правил (смена позиций).

## Ключевые концепции

### 1. Тестирование компонентов (Component Testing)
Мы использовали **Vitest** и **Vue Test Utils** для реализации подхода TDD. 
**Mental Model**: Думайте о компоненте как о "черном ящике". Мы тестируем входные данные (props) и события (emits), а также то, как внутреннее состояние отражается в DOM.
- *Урок*: Написание тестов до реализации помогло нам спроектировать чистый API для `MatchRecordingForm`, не отвлекаясь на стили.

### 2. Реактивное состояние и вычисляемые свойства
Мы применили `reactive` для управления формой и `computed` для производных данных.
- **Декларативный подход**: Вместо того чтобы вручную менять список доступных игроков при каждом выборе, мы создали вычисляемое свойство `selectedIds`. Это гарантирует, что UI всегда синхронизирован с данными.
- **Mandatory Swap**: Логика смены позиций для второй игры была реализована как производная от выбора в первой игре. Это исключает возможность рассинхронизации данных при навигации "назад-вперед".

### 3. Рефакторинг после Deep Review
В ходе глубокой проверки мы выявили и исправили несколько важных проблем:
- **Типизация**: Использование защитников типов (type guards) в фильтрации: `.filter((id): id is number => id != null)`.
- **Архитектура**: Перенос логики из `watch` в `computed` для обеспечения однонаправленного потока данных (unidirectional data flow).

## Заключение
Фаза 3 показала, как важно сочетать автоматизированное тестирование с ручной верификацией в "песочнице" (sandbox). Создание временных отладочных страниц (`DevRecordingView`) позволяет быстро итерировать сложные UI-компоненты до их интеграции в основное приложение.
