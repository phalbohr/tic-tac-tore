# Фаза 3: Frontend - Match Recording UI
## Задача: Implement Multi-Game Scoring with Position Swap Enforcement

### Введение
В этой задаче мы реализуем основной процесс ввода счета матча. Особенность Doubles (2v2) в настольном футболе — необходимость смены позиций (Нападающий/Защитник) между первой и второй игрой. Мы должны гарантировать, что система принудительно меняет позиции игроков во второй игре.

### Архитектурные решения
Мы продолжим использовать **Vue 3** и добавим состояние для управления текущей игрой (Game 1, Game 2, Game 3). 
Основные концепции:
1. **Стейт-машина (State Machine)**: Форма будет переходить между состояниями: "Выбор игроков" -> "Счет игры 1" -> "Счет игры 2" -> "Счет игры 3 (если нужно)".
2. **Валидация бизнес-правил**: Проверка обязательной смены позиций во второй игре будет реализована на уровне логики компонента.

### Ход выполнения

#### 1. Подготовка (Red Phase)
Тесты должны покрывать:
- Переход к вводу счета после выбора игроков.
- Обязательную смену позиций для Game 2.
- Возможность завершения матча после 2-х игр (если счет 2:0).
- Возможность выбора позиций для Game 3 (если счет 1:1).

#### 2. Реализация (Green Phase)
В компоненте `MatchScoring.vue` была реализована следующая логика:
- **Управление состоянием**: Используется массив `games` для хранения результатов до 3-х игр.
- **Автоматическая смена позиций**: При выборе позиций в Game 1, система автоматически рассчитывает "зеркальные" позиции для Game 2.
- **Динамический интерфейс**: Кнопки "Next Game" и "Finish Match" появляются только тогда, когда это логически обосновано (например, "Finish" доступен при счете 2:0 или после 3-й игры).

#### 3. Рефакторинг и оптимизация
После код-ревью были внесены улучшения:
- **Производные данные**: Вместо использования `watch` для синхронизации позиций в Game 2, теперь они вычисляются на лету. Это более "реактивный" подход, уменьшающий вероятность ошибок синхронизации.
- **Чистые функции**: Логика определения игрока по роли вынесена в чистую функцию, что упростило шаблон.
- **Валидация на уровне HTML**: Добавлены ограничения `min="0"` для полей ввода счета.

### Извлеченные уроки
- **Реактивность vs Наблюдатели**: В Vue 3 часто лучше использовать `computed` свойства для производных данных, чем `watch`. Это делает поток данных в приложении более предсказуемым (unidirectional data flow).
- **UX в спортивных приложениях**: Принудительная смена позиций в интерфейсе помогает игрокам не забыть об этом правиле в реальности, что критично для честной игры.
