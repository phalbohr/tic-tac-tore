# Deep Review Report

**Date:** 2024-03-21
**File reviewed:** src/main/java/com/tictactore/service/StatisticsService.java

## 01-arch-design-review.md

- **MEDIUM** — Сервис напрямую обращается к `SecurityContextHolder.getContext().getAuthentication()`. Это затрудняет тестирование и делает сервис зависимым от глобального состояния безопасности. Рекомендуется передавать данные пользователя (ID или объект) как параметр метода.
- **LOW** — Метод `calculateStatsFromMatches` дублирует логику, которая уже реализована в нативных SQL запросах репозитория. Хотя это может быть полезно для тестов или превью, это создает риск расхождения в расчетах при изменении логики (например, округления Win Rate).
- **LOW** — Использование `AtomicInteger` для ранжирования в `getLeaderboard` в памяти после получения данных из БД. Если БД возвращает данные с определенной сортировкой, это работает, но обычно ранг — это часть бизнес-логики, которую лучше вычислять в запросе (особенно если есть игроки с одинаковым Win Rate).

## 02-functionality-reliability-review.md

- **HIGH** — Метод `calculateStatsFromMatches` не проверяет статус матча (`MatchStatus.CONFIRMED`). Согласно спецификации (2.3/3.0), статистика должна учитывать только подтвержденные матчи. Если этот метод используется для расчета реальной статистики, он может вернуть некорректные данные, включая черновики и ожидающие одобрения матчи.
- **MEDIUM** — Метод `calculateSinceDate` использует `LocalDateTime.now()`. Без использования `Clock` (Dependency Injection) это делает логику расчета периодов времени труднотестируемой и зависимой от системного времени сервера.
- **MEDIUM** — В `getLeaderboard` отсутствует обработка ситуации, когда `minMatches` равен null. Хотя в SQL он используется как `:minMatches`, если передать null, поведение `HAVING COUNT(*) >= null` в PostgreSQL может привести к пустому результату вместо игнорирования фильтра.
- **LOW** — Округление Win Rate в `calculateWinRate` (Java) и в SQL запросах может слегка различаться из-за особенностей `double` и `NUMERIC`. В Java используется `Math.round(val * 10000.0) / 100.0`, а в SQL `ROUND(..., 2)`. Рекомендуется унифицировать способ расчета.

## 03-secure-code-review.md

- **LOW** — Прямое использование `SecurityContextHolder` в сервисном слое. Это не является критической уязвимостью, но считается плохой практикой (Insecure Direct Object Reference (IDOR) может возникнуть, если логика получения "текущего пользователя" будет случайно использована для действий от имени другого пользователя в будущем без явной проверки прав).

## 04-performance-review.md

- **HIGH** — В `findLeaderboard` используется `UNION ALL` по таблице `matches` четыре раза (для каждой роли каждого игрока). При большом количестве матчей это может привести к полному сканированию таблицы (Full Table Scan) четырежды. Рекомендуется оптимизировать запрос, используя один проход по таблице с `CASE WHEN`.
- **MEDIUM** — Метод `calculateStatsFromMatches` принимает `List<Match>` и итерируется по нему в памяти. Если список матчей будет очень большим (например, история игрока за несколько лет), это может привести к высокому потреблению памяти и задержкам. Лучше выполнять агрегацию на стороне БД.

## 05-test-review.md

- **CRITICAL** — В коде отсутствуют проверки на `null` для входящих параметров в методах `getPersonalStats` и `getH2HStats` (хотя они используют `getCurrentUser()`, который бросает исключение). Однако, отсутствие модульных тестов, подтверждающих корректность сложных SQL-запросов (особенно `H2HStats` и `findLeaderboard`), является высоким риском.
- **MEDIUM** — Сложная логика `calculateSinceDate` не покрыта тестами с фиксированным временем, что может привести к багам "граничных условий" (например, в конце года/месяца).

## 06-clean-code-review.md

- **MEDIUM** — Нарушение DRY: Логика вычисления "кто выиграл" и "какую роль занимал игрок" разбросана между методами `Match` (isWinner, isAttacker...) и нативными SQL запросами в `MatchRepository`. Любое изменение в правилах игры (например, введение 3v3) потребует обновления кода во многих местах.
- **LOW** — Магические строки в SQL запросах (например, `'CONFIRMED'`, `'TEAM_A'`). Рекомендуется использовать константы или параметры.
- **LOW** — `calculateSinceDate` возвращает фиксированную дату `2000-01-01` для `ALL_TIME`. Это "магическая дата", лучше использовать минимально возможную дату в БД или вообще убрать условие `created_at >= ...` из запроса для этого случая.

## 07-style-automation-review.md

- **LOW** — Использование `var` уместно, но в некоторых местах (например, в SQL проекциях) это может скрывать типы данных, возвращаемых из БД, что затрудняет чтение кода без IDE.

## 08-documentation-review.md

- **MEDIUM** — Публичные методы `getLeaderboard`, `getPersonalStats` и др. не имеют JavaDoc, описывающего ожидаемое поведение фильтров (например, включен ли конечный момент времени в период).

## 09-nitpick-review.md

- **LOW** — `Objects.requireNonNull(type, "type is required")` — сообщения об ошибках написаны на английском, в то время как проект может требовать локализацию или единообразие.
- **LOW** — Лишние пустые строки в некоторых методах и отсутствие пробелов вокруг некоторых операторов в SQL (хотя SQL внутри строк — это всегда компромисс).

## 10-logging-security-review.md

- **LOW** — В методе `getCurrentUser()` бросается исключение с email пользователя (`ResourceNotFoundException`). Если это исключение логируется или выводится пользователю, это может привести к раскрытию email (хотя в данном случае это email текущего аутентифицированного пользователя).

## 11-logging-review.md

- **CRITICAL** — Полное отсутствие логирования. В сервисе, выполняющем сложные расчеты статистики, критически важно логировать начало операций, параметры фильтрации и возможные ошибки (кроме стандартных исключений).

## 12-logging-error-handling-review.md

- **MEDIUM** — `IllegalStateException` в `getCurrentUser()` при отсутствии контекста аутентификации. Это техническая ошибка, которая должна быть обработана более изящно (например, через 401 Unauthorized), а не приводить к 500 Internal Server Error.

## 13-log-retention-review.md

- **LOW** — В коде нет логики, связанной с ротацией или хранением логов (так как логирования нет совсем).

## Tell-Don't-Ask Implementation Analysis

- **ПОЛОЖИТЕЛЬНО**: Использование методов `m.isWinner(player)`, `m.isAttacker(player)`, `m.isDefender(player)` в `calculateStatsFromMatches` является хорошим примером Tell-Don't-Ask. Сервис не запрашивает внутреннее состояние матча (кто был в какой команде), а просит сам объект матча ответить на вопрос о роли игрока.
- **ОТРИЦАТЕЛЬНО**: Нативные SQL запросы в `MatchRepository` по своей природе нарушают этот принцип, так как БД вынуждена "спрашивать" данные полей таблиц для выполнения агрегации. Однако, в контексте производительности для статистики это неизбежный компромисс.
- **УЛУЧШЕНИЕ**: Метод `calculateStatsFromMatches` мог бы быть еще более декларативным, если бы агрегация происходила внутри объекта, представляющего коллекцию матчей или в самом объекте `User`.
