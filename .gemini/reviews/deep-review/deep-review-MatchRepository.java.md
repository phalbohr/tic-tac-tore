# Deep Review Report

**Date:** 2024-03-24
**File reviewed:** src/main/java/com/tictactore/repository/MatchRepository.java

## 01-arch-design-review.md

- **MEDIUM** — Интерфейс репозитория содержит сложную бизнес-логику в виде нативных SQL-запросов. Хотя это допустимо для оптимизации производительности (согласно спецификации 3. Non-Functional Requirements), это затрудняет поддержку и тестирование. Рекомендуется рассмотреть возможность выноса сложных расчетов во View или материализованные представления в БД, если объем данных значительно вырастет.
- **LOW** — Репозиторий смешивает простые методы поиска по статусу и сложные агрегационные запросы для статистики. С точки зрения SRP, можно было бы разделить на MatchRepository и MatchStatisticsRepository.

## 02-functionality-reliability-review.md

- **MEDIUM** — В методе `findLeaderboard` параметр `since` может быть `null`, если логика фильтрации по времени не обеспечит значение по умолчанию. В текущем SQL-запросе это может привести к пустым результатам или ошибке, так как сравнение с NULL требует `IS NULL`.
- **LOW** — В запросе `findPendingApprovalsForUser` используется `DISTINCT`, что может быть избыточным при правильном использовании JOIN FETCH, но здесь оно оправдано из-за множественных JOIN. Однако это может скрыть ошибки дублирования данных в будущем.

## 03-secure-code-review.md

- **LOW** — Прямое использование строк в `ROUND(CAST(ps.wins * 100.0 / ps.total_matches AS NUMERIC), 2)` безопасно (SQL Injection отсутствует, так как используются параметры `:since`, `:type` и т.д.), но стоит убедиться, что входные данные для `type` и `since` валидируются в сервисном слое.
- **LOW** — Возвращение `playerId` как `String` в проекции (`CAST(u.id AS VARCHAR)`) является хорошей практикой для фронтенда, чтобы избежать проблем с точностью больших чисел, хотя UUID обычно передается как строка без проблем.

## 04-performance-review.md

- **HIGH** — Методы `findLeaderboard` и `findH2HStats` используют по 4 `UNION ALL` к таблице `matches`. При росте таблицы до сотен тысяч записей, сканирование таблицы 4 раза за один запрос станет узким местом.
    - *Рекомендация:* Создать индекс на `(status, created_at)` и на каждый столбец игрока (`team_a_attacker_id`, `team_a_defender_id` и т.д.). Еще лучше — пересмотреть схему данных для хранения участий игроков в отдельной таблице `match_participants`, что позволит считать статистику одним сканированием.
- **MEDIUM** — Метод `findLeaderboard` содержит сложный `countQuery`, который фактически повторяет логику основного запроса. Это удваивает нагрузку на БД при каждом запросе страницы.
- **MEDIUM** — Отсутствие индексов в спецификации. Для запросов типа `findAllConfirmedMatchesForUser` необходимы индексы по всем ID участников в таблице `matches`.

## 05-test-review.md

- **HIGH** — Сложные нативные запросы с CTE (`WITH`) требуют тщательного интеграционного тестирования на реальной БД (PostgreSQL), так как H2 может вести себя иначе или не поддерживать специфичные конструкции. Необходимо убедиться в наличии `StatisticsIntegrationTest.java`, покрывающего все граничные случаи (нет матчей, матчи за пределами периода `since`, матчи с разными статусами).

## 06-clean-code-review.md

- **MEDIUM** — Нативные запросы очень длинные и содержат дублирование логики (особенно блоки `UNION ALL`). Это нарушает принцип DRY.
    - *Рекомендация:* Использование SQL Views могло бы упростить репозиторий, оставив в нем только `SELECT * FROM player_participation_view`.
- **LOW** — Использование `com.tictactore.model.MatchStatus.CONFIRMED` внутри строки запроса JPQL — полное имя класса делает строку менее читаемой. Можно использовать параметры.

## 07-style-automation-review.md

- **LOW** — Непоследовательное использование форматирования в SQL блоках: где-то используется `UPPER CASE` для ключевых слов, где-то `lower case` (например, `as` vs `AS`). Стоит придерживаться одного стиля.

## 08-documentation-review.md

- **MEDIUM** — Отсутствует документация к сложным методам статистики. Непонятно, учитываются ли матчи 1v1 в общих лидербордах или только 2v2 (судя по коду, учитываются все, где заполнены соответствующие поля).
- **LOW** — Не указано, что `minMatches` является инклюзивным (`>= :minMatches`).

## 09-nitpick-review.md

- **LOW** — Лишние пробелы в начале строк внутри многострочных текстовых блоков (например, перед `WITH`).
- **LOW** — В `findH2HStats` закомментированные строки (`-- User in Team A...`) полезны, но можно было бы оформить их более структурированно.

## 10-logging-security-review.md

- **LOW** — Репозиторий не ведет логов (и не должен), но ошибки выполнения SQL могут содержать значения параметров. Необходимо убедиться, что на уровне драйвера БД или Spring не включено логирование значений параметров в продакшене.

## 11-logging-review.md

- **LOW** — Нет логов о времени выполнения тяжелых запросов. Для методов `findLeaderboard` и `findH2HStats` было бы полезно добавить мониторинг времени выполнения на уровне сервиса или через Spring AOP, так как они потенциально медленные.

## 12-logging-error-handling-review.md

- **MEDIUM** — При ошибках в нативном SQL (например, деление на ноль, если `total_matches` окажется 0 по какой-то причине, хотя `HAVING COUNT(*) >= 1` защищает от этого) выбросится общее `DataAccessException`. Стоит убедиться, что в сервисе есть специфичная обработка или проверки.

## 13-log-retention-review.md

- **LOW** — Данный файл не влияет на ротацию логов. Однако, если запросы будут генерировать много `Slow Query Logs` в БД, это может повлиять на место на диске. Необходимо настроить порог `long_query_time` в PostgreSQL.
