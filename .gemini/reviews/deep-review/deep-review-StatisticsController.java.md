# Deep Review Report

**Date:** 2026-03-24
**File reviewed:** src/main/java/com/tictactore/controller/StatisticsController.java

## 01-arch-design-review.md

### 🟡 Потенциальные риски

- **src/main/java/com/tictactore/controller/StatisticsController.java:23** — Противоречивость в использовании аннотаций. Контроллер реализует интерфейс `StatisticsApi`, где уже определены маппинги (например, `/leaderboard`). Однако для методов `/me` и `/h2h` аннотации `@GetMapping` продублированы в контроллере, а для `/leaderboard` — нет. Это создает путаницу и затрудняет поддержку. Рекомендуется придерживаться единого стиля: либо вынести все маппинги в интерфейс, либо дублировать их везде для наглядности (согласно стилю проекта в `MatchController`).

## 02-functionality-reliability-review.md

### 🔴 Критические ошибки

- **src/main/java/com/tictactore/controller/StatisticsController.java:47, 54** — Отсутствие фильтрации по периоду в персональной статистике и H2H. Согласно спецификации (AC 2.3), все статистические данные должны поддерживать фильтрацию (Weekly, Monthly, Yearly, All-time). В текущей реализации эндпоинты `/me` и `/h2h` не принимают параметр `TimePeriod`, что делает невозможным выполнение требований бизнеса.

### 🟡 Потенциальные риски

- **src/main/java/com/tictactore/controller/StatisticsController.java:54** — Отсутствие пагинации для Head-to-Head статистики. Спецификация (AC 2.4) явно требует наличие элементов управления пагинацией для таблиц H2H. Текущий метод возвращает `List<H2HStatsResponse>`, что не позволяет реализовать пагинацию на стороне сервера и может привести к проблемам при большом количестве оппонентов у игрока.

## 03-secure-code-review.md

### 🔵 Рекомендации по улучшению

- **src/main/java/com/tictactore/controller/StatisticsController.java:23** — Отсутствие явных проверок авторизации на уровне методов. Хотя безопасность может быть настроена глобально, использование `@PreAuthorize("isAuthenticated()")` или аналогичных аннотаций непосредственно в контроллере повышает защищенность кода (defense in depth) и делает намерения разработчика более явными.

## 04-performance-review.md

### 🟡 Потенциальные риски

- **src/main/java/com/tictactore/controller/StatisticsController.java:54** — Риск деградации производительности при росте данных. Без пагинации эндпоинт `/h2h` будет возвращать всех оппонентов, с которыми играл пользователь. Со временем это может привести к передаче больших объемов JSON и увеличению времени ответа. Рекомендуется использовать `Pageable` аналогично эндпоинту `/leaderboard`.

## 05-test-review.md

### 🟡 Потенциальные риски

- **src/test/java/com/tictactore/StatisticsIntegrationTest.java** — Недостаточное покрытие тестами граничных условий и бизнес-правил. В тестах отсутствуют проверки того, что:
    1. В статистику попадают только `CONFIRMED` матчи.
    2. Фильтр `minMatches` корректно отсеивает игроков в лидерборде.
    3. Фильтрация по времени (`TimePeriod`) корректно изменяет результаты.

## 06-clean-code-review.md

### 🔵 Рекомендации по улучшению

- **src/main/java/com/tictactore/controller/StatisticsController.java:41, 48, 55** — Избыточное логирование. Логи уровня `INFO` на каждый запрос могут засорять вывод в высоконагруженных системах. Рекомендуется перевести логи начала обработки запроса на уровень `DEBUG`, оставив на `INFO` только значимые события (например, ошибки или редкие бизнес-действия).

- **src/main/java/com/tictactore/controller/StatisticsController.java:38** — Использование `var` в контроллере. Хотя это допустимо в Java 21, в контроллерах иногда предпочтительнее явное указание типов для улучшения читаемости без использования IDE, особенно когда возвращается сложный тип данных вроде `Page<LeaderboardEntryResponse>`.

## 07-style-automation-review.md
